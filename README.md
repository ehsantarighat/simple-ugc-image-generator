# UGC Studio

Hyper-realistic, influencer-style product photography from a model reference,
a product reference, and a scene description. Powered by GPT Image 2.

This is the MVP of a production-grade web platform: real generations end-to-end,
not mock screens.

## Stack

- **Next.js 15** (App Router, Server Components, Server Actions)
- **TypeScript** strict, **Tailwind CSS v4**, **shadcn/ui** primitives (hand-vendored)
- **Supabase** — auth, Postgres, private object storage with RLS
- **OpenAI Node SDK** (`openai`) — GPT Image 2 via the `images.edit` endpoint
- **Zod** for validation, **Sharp** for upload preprocessing
- **Sonner** for toasts

## Features (MVP)

- Email/password authentication via Supabase Auth
- Reusable **Models** library (multi-image upload, edit, delete)
- Reusable **Products** library (multi-image upload, preservation notes, edit, delete)
- **Projects** with a full generation studio:
  - Pick model + product (selection is persisted)
  - Free-form scene description
  - All 8 photography controls (shot type, angle, lens, framing, lighting,
    authenticity, product prominence, aspect ratio) + 1–4 variations
- **GPT Image 2** generation pipeline — references uploaded, prompt rendered,
  outputs persisted, RLS enforced
- **Refinement** — re-shoot any output with a free-form instruction. Stored
  as a child image linked to its parent.
- **Gallery** — past runs, status badges, favorite, download
- **Download** — server-mediated, ownership-checked, streamed
- **Server-side prompt rendering** — users never see prompt complexity

## Quick start

### 1. Install

```bash
pnpm install
```

### 2. Supabase

1. Create a Supabase project at <https://supabase.com>.
2. In **Project Settings → API**, copy the URL, anon key, and service role key
   into a new `.env.local` (see `.env.example`).
3. Apply the SQL migrations in this order from `supabase/migrations/`:
   - `0001_init.sql` — tables, indexes, triggers
   - `0002_rls.sql` — row-level security policies
   - `0003_storage.sql` — the `ugc-assets` bucket + access policies
4. In **Authentication → URL Configuration**, set:
   - Site URL: `http://localhost:3000`
   - Additional redirect URL: `http://localhost:3000/auth/callback`
5. (Optional, recommended for local dev) In **Authentication → Providers →
   Email**, disable "Confirm email" so signup flows don't require an inbox.

See [`supabase/README.md`](supabase/README.md) for the detailed walkthrough.

### 3. OpenAI

Set `OPENAI_API_KEY` in `.env.local`. The default model is `gpt-image-2`;
override via `OPENAI_IMAGE_MODEL` if your account only has `gpt-image-1`.

### 4. Run

```bash
pnpm dev
```

Open <http://localhost:3000>.

## Architecture notes

- **Auth flow.** Supabase email/password. Server actions in
  `lib/actions/auth.ts` call `supabase.auth.signInWithPassword` /
  `signUp`. The middleware at `middleware.ts` refreshes the session on every
  request and gates protected routes.
- **Storage.** A single private bucket `ugc-assets` with paths namespaced by
  user id: `models/{userId}/{modelId}/`, `products/{userId}/{productId}/`,
  `generated/{userId}/{projectId}/{requestId}/`,
  `revisions/{userId}/{projectId}/{revisionId}/`. Storage RLS enforces that
  the second path segment must equal `auth.uid()` — defense-in-depth on top of
  the table-level RLS.
- **Upload pipeline.** `lib/services/asset-service.ts` validates MIME + size,
  re-encodes with `sharp` (max edge 2048, JPEG q88) to normalize EXIF and
  keep file sizes manageable, then uploads using the user's authenticated
  session so RLS stays in the loop.
- **Generation pipeline.** `lib/services/image-generation-service.ts` is the
  core service. It:
  1. Loads model + product + their reference images
  2. Builds a structured payload via `GenerationPayloadBuilder`
  3. Renders the final prompt (`renderPromptFromPayload`)
  4. Inserts a `generation_requests` row in `generating` state
  5. Downloads reference bytes, builds OpenAI File objects, calls
     `client.images.edit({ model, image, prompt, size, n })`
  6. Persists each output to storage + a `generated_images` row
  7. Updates the request to `completed` or `failed`
- **Refinement pipeline.** `lib/services/refinement-service.ts` re-runs the
  pipeline with the previous output added as a reference image and the user's
  refinement instruction layered onto the original prompt. Outputs are stored
  with `parent_image_id` pointing back to the source.
- **Status lifecycle.** `draft → queued → generating → completed | failed`.
  Today everything runs inline in the route handler. A background worker can
  consume `queued` rows without touching the public surface.
- **API surface.** Server actions are used wherever possible (RSC-friendly,
  no double round-trip). Route handlers are reserved for:
  - `POST /api/generate` — start a generation (long-running)
  - `GET  /api/generate?id=...` — poll status
  - `POST /api/refine` — refinement
  - `GET  /api/download?id=...` — server-mediated download
- **Prompting.** The user never sees prompt internals. Internally we build
  the structured payload first and then render to natural language, with a
  fixed set of anti-artificial negative constraints baked in (CGI sheen,
  malformed hands, label gibberish, etc.).

## How generation works (under the hood)

User picks a model + product, types "Morning routine in a softly lit bathroom,
holding the serum bottle near her cheek, smiling at the mirror." and the
controls `candid_lifestyle / eye_level / smartphone_portrait /
medium_close_up / soft_window_light / natural_influencer / balanced / 4:5 / 4`.

The server then sends a `client.images.edit({ ... })` call to GPT Image 2
with:

- Up to 5 model reference images + 5 product reference images attached
- A rendered prompt that names every control and bakes in a "must look like
  a real photo" framing, identity-preservation language, product-preservation
  language, and ~9 anti-CGI negative constraints

Each of the requested variations comes back as base64 PNG, gets uploaded to
private storage at `generated/{userId}/{projectId}/{requestId}/{n}.png`, and
gets a `generated_images` row that the UI then resolves via signed URLs
(15-minute expiry).

## Known limitations and next improvements

Shipped:

- ✅ End-to-end generation with reference preservation
- ✅ Refinement workflow with parent linking
- ✅ Reusable model and product libraries with multi-image upload
- ✅ All 8 photography controls
- ✅ Server-mediated downloads
- ✅ Status lifecycle ready for a future queue worker

Deferred:

- ⏳ **Quality review** — schema is there (`generated_images.review_json`),
  service is stubbed in `quality-review-service.ts`. Implementing it well
  requires a vision model call; out of MVP scope.
- ⏳ **Background queue** — generation runs synchronously inside the route
  handler. With Fluid Compute that's fine for the MVP, but a `queued`
  worker would be cleaner.
- ⏳ **Activity logs UI** — the table exists, no UI consumes it yet.
- ⏳ **Settings editing** — profile fields are read-only.
- ⏳ **Tests** — no automated tests in the MVP.
- ⏳ **Scene templates and batch generation** — out of MVP scope.
- ⏳ **Polling for very long generations** — `GET /api/generate?id=...`
  exists but the UI runs the POST synchronously today.
